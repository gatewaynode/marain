  Mental model — the canonical transpiler pipeline

  source bytes (.lat)
    ─▶ lexer    ─▶ tokens    (each carrying a Span)
    ─▶ parser   ─▶ AST       (each node carrying a Span)
    ─▶ [lower]  ─▶ IR        (Stage 2 only; v0.1 parses straight to emit-ready AST)
    ─▶ emitter  ─▶ Rust source (a String)
    ─▶ driver   ─▶ write shim crate, invoke cargo, forward stdout/stderr

  Every box is hand-rolled in Rust (no external crates per §9). For Stage-1 hello-world (dic "salve, munde".) the entire
  walk is small — likely ~400 LOC of working code. The shape matters more than the content because every later feature
  reuses the same boxes.

  3. Error model — per-stage error enums vs. one flat facade
  4. Lexer — token kinds for hello-world; module split to respect the 500-LOC ceiling
  5. Parser + AST — recursive descent; AST shape (and the one Stage-2 hook)
  6. Codegen + cargo shim — emission strategy + on-disk layout of the generated project
  7. CLI + driver — hand-rolled arg parsing; marain build / marain run
  8. Testing harness — behavioral end-to-end + per-phase unit tests

  Concerns I'm carrying forward (none block v0.1, all worth a beat)

  - (α) Stage 2's parser is a different animal. Hand-rolled recursive descent doesn't extend to free-word-order; Stage 2
  will likely rewrite the parser entirely (Earley / GLR / case-driven assembly). The Stage 1 parser is a deliberate
  throwaway — but our AST nodes should probably carry an optional inflection slot now so the node types survive the Stage
  2 transition.
  - (β) The 500-LOC ceiling bites the lexer first. Tokenizers grow fast as the keyword table accretes. Plan the split now
  (scanner.rs / keywords.rs / strings.rs), not at the 480-LOC mark.
  - (γ) The vendored Variabile runtime (§5). Not v0.1, but the codegen architecture needs an obvious injection point.
  Three patterns: inline-at-codegen / runtime-crate-import / emit-support-module-verbatim-into-shim. I'd lean (c) for the
  self-supporting story.
  - (δ) Hand-rolled CLI parsing (no clap). ~50 LOC and easy to get subtly wrong on edge cases (--help, --version, unknown
  flags). Worth a dedicated cli/args.rs, not freestyle in main.rs.
  - (ε) Test strategy. Behavioral end-to-end is required (PRD §7). Golden-diff on emitted Rust text is fragile to
  whitespace/formatting; treat it as a tripwire only. Per-phase unit tests + the e2e do most of the work.

  Mental model — the canonical transpiler pipeline

  source bytes (.lat)
    ─▶ lexer    ─▶ tokens    (each carrying a Span)
    ─▶ parser   ─▶ AST       (each node carrying a Span)
    ─▶ [lower]  ─▶ IR        (Stage 2 only; v0.1 parses straight to emit-ready AST)
    ─▶ emitter  ─▶ Rust source (a String)
    ─▶ driver   ─▶ write shim crate, invoke cargo, forward stdout/stderr

  Every box is hand-rolled in Rust (no external crates per §9). For Stage-1 hello-world (dic "salve, munde".) the entire
  walk is small — likely ~400 LOC of working code. The shape matters more than the content because every later feature
  reuses the same boxes.

  Design rounds I see ahead

  1. Skeleton — crate layout + initial module decomposition ← this round
  2. Span & source-map — how source position is carried through every later phase
  3. Error model — per-stage error enums vs. one flat facade
  4. Lexer — token kinds for hello-world; module split to respect the 500-LOC ceiling
  5. Parser + AST — recursive descent; AST shape (and the one Stage-2 hook)
  6. Codegen + cargo shim — emission strategy + on-disk layout of the generated project
  7. CLI + driver — hand-rolled arg parsing; marain build / marain run
  8. Testing harness — behavioral end-to-end + per-phase unit tests

  Concerns I'm carrying forward (none block v0.1, all worth a beat)

  - (α) Stage 2's parser is a different animal. Hand-rolled recursive descent doesn't extend to free-word-order; Stage 2
  will likely rewrite the parser entirely (Earley / GLR / case-driven assembly). The Stage 1 parser is a deliberate
  throwaway — but our AST nodes should probably carry an optional inflection slot now so the node types survive the Stage
  2 transition.
  - (β) The 500-LOC ceiling bites the lexer first. Tokenizers grow fast as the keyword table accretes. Plan the split now
  (scanner.rs / keywords.rs / strings.rs), not at the 480-LOC mark.
  - (γ) The vendored Variabile runtime (§5). Not v0.1, but the codegen architecture needs an obvious injection point.
  Three patterns: inline-at-codegen / runtime-crate-import / emit-support-module-verbatim-into-shim. I'd lean (c) for the
  self-supporting story.
  - (δ) Hand-rolled CLI parsing (no clap). ~50 LOC and easy to get subtly wrong on edge cases (--help, --version, unknown
  flags). Worth a dedicated cli/args.rs, not freestyle in main.rs.
  - (ε) Test strategy. Behavioral end-to-end is required (PRD §7). Golden-diff on emitted Rust text is fragile to
  whitespace/formatting; treat it as a tripwire only. Per-phase unit tests + the e2e do most of the work.

  Round 1 proposal — skeleton

  A single crate, with lib.rs doing the work and main.rs as the thin CLI shim. Cargo will treat them as one library + one
  binary, both named marain:

  marain/
    Cargo.toml
    Cargo.lock
    rust-toolchain.toml          # pin stable (TBD: which 1.8x?)
    src/
      lib.rs                      # crate root; module declarations + re-exports
      main.rs                     # thin CLI shim — parses args, dispatches, forwards exit codes
      source.rs                   # SourceFile + SourceMap (owns file bytes + line-break index)
      span.rs                     # Span newtype (u32 start, u32 end)
      error.rs                    # MarainError facade + Diagnostic rendering
      lexer/
        mod.rs                    # public API
        scanner.rs                # byte/char-level scanning
        keywords.rs               # the keyword/no-punct-macro table
      token.rs                    # TokenKind enum + Token struct
      parser/
        mod.rs                    # recursive descent driver
        grammar.rs                # one fn per production (small for v0.1)
        keywords.rs               # the keyword/no-punct-macro table
      token.rs                    # TokenKind enum + Token struct
      parser/
        mod.rs                    # recursive descent driver
        grammar.rs                # one fn per production (small for v0.1)
      ast.rs                      # AST node types (every node carries Span)
      emit/
        mod.rs                    # AST → Rust string entrypoint
        rust.rs                   # the actual templated emission
      shim/
        mod.rs                    # generates the sibling Cargo project (Cargo.toml + src/main.rs)
      cli/
        mod.rs                    # subcommand dispatch
        args.rs                   # hand-rolled arg parser
        build_cmd.rs              # `marain build` body
        run_cmd.rs                # `marain run` body
    tests/
      hello_world.rs              # e2e: build hello.lat, run it, assert "salve, munde" on stdout
    docs/
      lexicon.md                  # (PRD-referenced; will accrete as keywords resolve)

  Why single crate, not a workspace? A workspace (separate crates/marain/ library + crates/marain-cli/ binary) buys a hard module boundary at the cost of a second Cargo.toml, a workspace root, and
  more file churn. For v0.1 the boundary is unnecessary — lib.rs + main.rs in one crate satisfies CLAUDE.md's "thin CLI shim" requirement (the lib is the testable / embeddable surface; the bin is
  ~50 LOC of arg dispatch). We can graduate to a workspace later if Stage 2's LSP wants to depend on the library independently of the CLI.

  Why split lexer/ and parser/ into directories already? Per concern (β) — they're the modules most likely to brush the 500-LOC ceiling first. Starting them as directories lets us add strings.rs,
  numbers.rs, etc. without restructure pain later.

  Why separate source.rs and span.rs? Spans are tiny and pervasive (every token, every AST node). The SourceMap is the heavyweight thing that owns the file bytes and the line-break index for
  diagnostic rendering. Different change rates, different dependency profiles, different blast radius.

  Why shim/ as its own module? Generating a Cargo project on disk is a chunky responsibility (template strings for Cargo.toml, layout decisions, eventually the vendored runtime injection per
  concern γ). It deserves its own home from day one.

  One genuine fork worth deciding now: where does the generated cargo shim live on disk?

  This is user-visible and changing it later means breaking everyone's mental model.
