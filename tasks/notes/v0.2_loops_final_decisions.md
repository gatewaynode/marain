  B. Type system scope for v0.2

  The PRD §4.11.1 commitment is just the syntax: (<sigiled-name>: <Tipus>) dat <ReturnTipus>. It says nothing about what <Tipus>
  accepts or how Tipus lowers to Rust. Three live options:

  B-1. Minimal closed alphabet. Hard-code Sermo and Numerus as keyword-class type names. Lexer adds two Keyword::Type* variants;
  parser only accepts those in type position. Emit maps Sermo→String, Numerus→i64. Closed set; every future type needs
  PRD+lexer+emit edits.

  B-2. Open pass-through. Accept any PascalCase PlainIdent in type position. Emit it verbatim. If Sermo isn't a real Rust type,
  rustc errors. PRD §5 already commits to pass-through rustc errors as a v0.1+ wart.

  B-3. Hybrid (recommended). Parser accepts any PascalCase ident in type position. Emitter holds a small translation table — at
  v0.2 it's two entries (Sermo→String, Numerus→i64). Names not in the table emit verbatim. Lexer doesn't change. This gives users
  canonical Marain types today and a forward path when structura lands (user-defined types just pass through).

  Bounded sub-decision in any option: generics syntax (Agmen<T>). PRD §4.11.6 defers generics to v0.3+, so the parser rejects < in
  type position with an explicit "deferred" error. Worth committing in R12's framing so nothing leaks.

  My pick: B-3. Lowest commitment, no PRD edit needed, doesn't constrain v0.3.
  
Answer (B): Ok, that makes a lot more sense.  I agree B-3 makes sense, we can punt on additional types until they are needed.

  C. Lowering pass between parser and emitter

  Current v0.1 shape: parse() → Module → emit(). One AST. A lowering pass would split that into parse() → RawAst → lower() → Ast →
  emit() — an IR layer where syntactic sugar normalizes to a canonical form before emission.

  What v0.2 actually needs lowering for: almost nothing. Walking the constructs:
  - aliter si → else if chain: parser-level (recognize the two-token sequence, build one IfElse node with Else::If(...)).
  - maior quam, non aequat, divisus per: parser-level phrase recognition (ARCHITECTURE §6.3 already commits to this).
  - semper :, nihil., 0..10: 1:1 emit, no transform.
  - pro ^i in <iter> :: 1:1 to Rust for.

  Three options:

  C-1. Build lowering layer now. Scaffolds the Stage 2 seam early. Cost: every AST variant duplicated across RawAst/Ast, mostly
  identity transforms. Pure YAGNI tax.

  C-2. Defer (recommended). Parser produces Ast directly, same as v0.1. ARCHITECTURE §7.8 already documents the Stage 2 hook: when
  free-word-order lands, rename Ast→RawAst, introduce new Ast, fill in lowering. Bounded lift. No v0.2 cost.

  C-3. Single desugar function (no IR split). Add desugar(Module) -> Module between parse and emit. Acts as no-op for most v0.2
  constructs. Buys nothing concrete; introduces a layer with no first user.

  My pick: C-2. The constructs don't motivate the pass. The Stage 2 seam stays documented; we cut it when something actually needs
  cutting.
  
Answer (C): Also agree, deferring (C-2) seems the quickest route at little future cost.

  D. Comment syntax (carry-over η)

  Today: no comment syntax exists. Lexer treats unrecognized bytes as LexError::UnexpectedChar. So zero .lat files can carry
  comments — tolerable for hello-world, painful as soon as v0.2 functions land.

  Four sub-questions:

  D-1. Marker shape. Candidates:
  - // (Rust). Zero cognitive cost for a Rust-speaking author. Doesn't eat the Latin keyword namespace.
  - # (Python/shell). One char; visually quiet; collides with nothing.
  - Latin word (nota, commentum). Aesthetically on-brand but eats keyword space and creates a "comment ends at what" problem
  (newline? period? — both fight §4.8).
  - -- (SQL/Latin academic). Conflicts visually with Minus Minus; requires lookahead.

  D-2. Line + block + doc? Rust offers all three. Line (//) covers 95% of need. Block (/* */) opens nesting questions. Doc (///)
  needs a doc story, which v0.2 doesn't have (PRD §8 non-goals).

  D-3. Lexer impact. Comments are lookahead + consume-to-EOL + emit-no-token. New file lexer/comments.rs (~30 LOC). Indent tracking
   unaffected (comments live inside lines, between Period and newline).

  D-4. PRD gating. Of B/C/D, this is the only one needing a PRD edit first. New §4.12 "Comments" with marker, EOL-terminated
  semantics, deferred block/doc. Plus §11 marks η resolved.

  My pick: // line comments only; PRD §4.12 amendment as the first task of the v0.2 implementation phase, before R9. Reasoning:
  zero keyword namespace cost, familiar to the audience, single small lexer addition, and it unblocks human-readable fixtures for
  every round that follows. Block/doc deferred until a real Marain program demonstrates pain.

Answer(D): Agreed, we need some sort of comment syntax.  We can go for just line comments right now "//", but we definitely need space for multi line comments in the future "/* */".
